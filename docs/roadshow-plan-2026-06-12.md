## 4. Demo 演示脚本（90 秒，固定话术）

> 讲一个连贯小故事：安排明天的会议 → 查看 → 改时间 → 删除。全程用练熟的句子。

| 步骤 | 你的操作 | 你说的话术（对麦克风） | 解说词（对评委） |
|------|----------|------------------------|------------------|
| 1 添加 | 按住麦克风 | 「明天下午三点开会」 | 「注意它不是死板回复，而是有温度的自然语言」 |
| 2 反馈 | 松开，等 TTS | —— | 「ASR + 大模型解析 + 语音播报，全中文管线」 |
| 3 查询 | 按住麦克风 | 「明天有什么安排」 | 「日历上对应日期出现了圆点标记」 |
| 4 修改 | 按住麦克风 | 「把明天的开会改到下午四点」 | 「修改前会弹确认，避免误操作」 |
| 5 删除 | 按住麦克风 | 「删除明天的开会」 | 「删除同样需要确认，数据全部存在本地 SQLite」 |

**收尾一句**：「整个过程不用打字、不用点按钮，纯语音完成增删改查，数据不出本机。」

## 5. 架构讲解要点（答辩用）

```
React Native 前端 (本地 SQLite 存储)
        │  REST API
        ▼
FastAPI 后端（无状态）
  讯飞 ASR ──► DeepSeek NLU ──► 讯飞 TTS
  语音→文字     文字→意图        文字→语音
```

**必须强调的两点（直接对应「代码质量 40%」）：**
- **隐私**：后端无状态，不存任何日程，事件数据只在用户设备本地 SQLite。
- **原创**：自写原生 AudioModule（替代不兼容的第三方音频库）、自研意图解析与事件管理逻辑，第三方库仅用于网络/UI/数据库等基础能力。

## 6. 评委 QA 预案

| 可能的提问 | 建议回答要点 |
|------------|--------------|
| 为什么后端做成无状态？ | 隐私优先，数据留在本地；后端只做语音与 NLU，易横向扩展、易测试。 |
| 为什么选讯飞 + DeepSeek？ | 讯飞中文 ASR/TTS 成熟；DeepSeek 结构化 JSON 输出稳定、性价比高、响应快（已切 v4flash）。 |
| 延迟多少？怎么优化的？ | 全链路约数秒；优化点：TTS 不在主管线重复合成、复用 httpx 连接池、调整 ASR 发送帧率。 |
| 怎么保证识别/解析准确？ | Prompt 内置当前日期与相对时间规则（明天/下周三/这周末）、模糊时间默认值；空/噪音输入入口拦截，避免编造。 |
| 怎么处理「没听清/乱说」？ | 空文本与短噪音直接返回无意图并友好提示，不凭空造日程。 |
| 有没有做流式/实时？ | 有 WebSocket 实时音频分支（`feat/*-websocket-voice`），主分支用稳定的请求-响应链路保证可靠性。 |
| 后续规划？ | 日期范围查询「看看这周安排」、一句话拆多事件、习惯/日常任务（参考 `docs/competitive-analysis.md`）。 |

## 附：本次新增改动记录

### 稳定性修复
- `server/services/nlu.py`：`parse_intent` 入口新增空文本 / 短噪音拦截，杜绝现场凭空编造日程。

### 性能优化（总计省约 2-3 秒）
| 改动 | 文件 | 预估收益 |
|------|------|----------|
| httpx 连接池复用 | `server/services/nlu.py` | ~300-500ms（省掉每次 TCP+TLS 握手）|
| TTS 预合成缓存 | `server/main.py` | ~1-2s（主管线返回时异步启动合成，前端请求时直接命中缓存）|
| ASR 帧间隔 0.01→0.005 | `server/services/xf_asr.py` | ~200ms（加速音频帧传输）|

### 优化后预估全链路时间（HTTP 模式）
```
松开录音 → 上传(0.3s) → ASR(1-1.5s) → NLU(0.5-2s) → 返回文本(0.1s)
                                                        ↓ (同时启动)
前端处理意图(0.1s) → 请求 TTS → 缓存命中直接返回 → 播放
```

### WebSocket 流式模式（核心提速）
```
按住说话时: 音频实时流向后端 → 后端实时转发讯飞 ASR
松开时: ASR 已在处理中，~0.3s 返回最终结果 → NLU(0.5-2s) → 返回
                                                    ↓ (同时启动)
                                              TTS 预合成(缓存命中)
```
- 用户看到文字反馈：**松开后 ~1-3 秒**（之前 10+ 秒）
- 用户听到语音回复：文字反馈后**几乎无额外等待**
