# VocaCal 产品演进方案

> 定位：从"能用的语音日历 demo"进化为"中文世界最好用的语音优先日历产品"
> 前提：不限时间、AI coding 开发（实现成本低，规划可以大胆）、视觉层已完成 iOS HIG 重做（系统蓝 #007AFF / 分组灰底 / inset 列表 / Large Title）
> 评分参照：完整度+创新 40% ｜ 代码质量+开发过程 40% ｜ 演示表达 20%

---

## 一、产品定位与差异化价值主张

### 一句话定位

**「说一句话，日程就排好」—— 专为中文口语设计的语音优先日历，数据不出本机。**

### 三层差异化（为什么评委/用户会觉得"不一样"）

| 层 | 价值主张 | 竞品现状 |
|----|---------|---------|
| **语音体验层** | 流式 ASR + 松开 1-3 秒出结果 + 温情 TTS 回复，全链路中文优化 | Google/Fantastical 主攻英文；小爱绑定硬件生态；没有独立的中文语音优先日历 App |
| **信任层** | 每次操作有"理解了什么"的透明展示 + 确认/撤销/纠错路径，AI 不凭空编造 | 竞品普遍"黑盒"，用户不知道 AI 理解了什么，不信任就不用 |
| **隐私层** | 后端无状态，日程只存本地 SQLite | 所有云日历都要上传数据；这是答辩时的硬差异点 |

### 产品人格

不做冷冰冰的工具，做"有温度的日程小助手"：回复俏皮简短（"安排上啦！"）、今日概览有人情味（"今天很忙碌，注意休息"）、日程洞察主动关怀。这是中文竞品完全空白的体验带。

## 二、核心体验短板诊断（基于现有代码）

| # | 短板 | 代码依据 | 影响 |
|---|------|---------|------|
| 1 | **一句话只能一个事件** | `nlu.py` prompt 只返回单个 JSON 对象；`intent.ts` 的 `NLUResult` 是单事件结构 | "明天开会后天体检"只解析出一个，违背口语习惯，是最大的能力天花板 |
| 2 | **查询只支持单日** | `calendarIntentService.ts` 的 query 按单个 `date` 查 `getEventsByDate` | "看看这周安排"无法回答，语音日历的高频场景缺失 |
| 3 | **没有提醒通知** | 全仓无 notification 相关代码 | 加了日程不提醒 → 用户不会信任这个日历，闭环断裂 |
| 4 | **没有撤销** | `CalendarScreen.tsx` 删除/添加后无 undo 入口 | 语音误识别 + 不可撤销 = 用户焦虑，语音产品大忌 |
| 5 | **识别错了没有纠错路径** | `handleVoiceStop` 失败只显示 `statusMessage` 一行字 | "抱歉没理解"一句话打发，用户只能整句重说 |
| 6 | **无多轮对话** | WebSocket 会话一次性关闭（`cleanup()`），后端无会话状态 | "把那个改到后天"这类指代无法理解；缺少"时间没说→追问"的补全机制 |
| 7 | **状态反馈是纯文字** | `statusMessage` 单行 Text | 解析结果没有结构化卡片展示，"理解了什么"不透明 |
| 8 | **视图单一** | 只有月历+单日列表 | 没有周视图/议程视图，重度用户无法纵览 |
| 9 | **错误处理粗糙** | `apiService.ts` 无重试/超时配置；离线时等满 timeout 才报错 | 网络抖动一次就失败；无网络预检 |
| 10 | **无搜索** | `storageService.ts` 没有按关键词查询接口 | 日程多了找不到 |

## 三、功能演进路线图

### P0 — 决定产品成败的核心闭环（优先全部做完）

#### P0-1 一句话多事件拆分 ⭐ 最大差异化
- **用户问题**：口语习惯是一口气说完："明早健身、下午三点开会、晚上和老王吃饭"。
- **交互**：解析后展示一张"批量确认卡片"，列出 N 个待添加事件（可单个取消勾选），确认后一次性入库；TTS 回复"帮你安排了 3 件事"。
- **技术改动**：
  - NLU prompt 改为返回 `{"events":[...]}` 数组（向后兼容单事件）；
  - `intent.ts` 增加 `NLUMultiResult`；`calendarIntentService.ts` 增加 `applyIntents` 批量逻辑；
  - 新组件 `BatchConfirmCard`。
- **评分贡献**：创新性 ↑↑（滴答清单同款杀手级功能，语音场景更强）；Demo 必秀。
- **复杂度**：中（prompt + 数据结构 + 一个新组件）。

#### P0-2 日期范围查询（这周/下周/这个月/周末）
- **用户问题**："看看这周安排"是语音日历最高频查询。
- **交互**：返回按天分组的议程列表视图；TTS 总结"这周有 5 个安排，最忙的是周三"。
- **技术改动**：NLU prompt 增加 `date_range:{start,end}` 字段；`storageService.ts` 增加 `getEventsByDateRange`；前端新增按天分组的 AgendaList。
- **评分贡献**：功能完整度 ↑↑。
- **复杂度**：中低。

#### P0-3 事件提醒通知
- **用户问题**：不提醒的日历没人敢依赖（信任闭环的最后一块）。
- **交互**：添加事件默认提前 15 分钟本地通知（可在事件上调整）；通知点击直达当天视图。
- **技术改动**：接入 `@notifee/react-native`（需重新构建 APK）；`event.ts` 增加 `remindMinutes` 字段 + SQLite 迁移；增删改时同步注册/取消通知。
- **评分贡献**：功能完整度 ↑↑（"所有竞品标配"补齐）。
- **复杂度**：中（涉及原生依赖与重打包，安排在非路演敏感期）。

#### P0-4 撤销（Undo）
- **用户问题**：语音误识别不可避免，必须有"后悔药"。
- **交互**：任何添加/删除/修改后，状态条变成"已添加 ×1 ［撤销］"，5 秒内可一键撤回；语音说"撤销"也能触发。
- **技术改动**：`calendarIntentService.ts` 记录最近一次操作的逆操作；NLU 增加 `UNDO` 意图；状态条组件加按钮。
- **评分贡献**：交互流畅度 ↑、创新 ↑（语音撤销很出彩）。
- **复杂度**：低中。

#### P0-5 结构化解析反馈卡片（信任感核心）
- **用户问题**：用户不知道 AI"理解了什么"，黑盒导致不信任。
- **交互**：识别完成后，先弹出一张解析卡片：`📅 6月12日 ⏰ 15:00 📝 开会`，字段逐项高亮飞入（学 Fantastical），1.2 秒后自动落库或等用户确认。
- **技术改动**：新组件 `IntentPreviewCard` + 简单进场动画；`CalendarScreen` 在 `handleIntentResult` 前插入展示态。
- **评分贡献**：Demo 表现力 ↑↑（视频里最抓眼球的一幕）、信任感差异化。
- **复杂度**：低中（纯前端）。

### P1 — 让产品"完整"的体验层

| 功能 | 用户问题 | 交互/技术要点 | 评分贡献 | 复杂度 |
|------|---------|--------------|---------|--------|
| **多轮对话补全** | "明天开会"没说时间 → 不该直接落全天 | 后端返回 `need_clarify:{field:"time"}`，App 追问"几点开始？"并保持录音会话上下文（前端暂存半成品事件即可，后端仍无状态） | 创新 ↑↑ | 中 |

| **冲突检测增强** | 同时段重复安排 | 已有基础（`hasConflict`），升级为：解析卡片上直接标红冲突 + TTS 提醒 + 提供"仍要添加/换时间"选项 | 完整度 ↑ | 低 |
| **日程洞察** | 用户不知道自己一周的负荷 | 周查询时附加洞察："这周 3 天晚上有安排，注意休息"；本地规则即可，无需模型 | 创新 ↑↑（竞品空白） | 低 |
| **周视图/议程视图切换** | 单日列表纵览不足 | 顶部 segmented control：月/周/议程；周视图复用 range 查询 | 完整度 ↑ | 中 |
| **关键词搜索** | 找不到旧日程 | `storageService` 加 LIKE 查询 + 顶部搜索框；语音"找一下上次的面试"也走它 | 完整度 ↑ | 低 |
| **网络预检 + 重试** | 离线按麦克风等超时 | 录音前 ping `/api/health`（800ms 超时）；axios 加一次自动重试 | 代码质量 ↑ | 低 |
| **离线降级** | 无网时完全不可用 | 文字输入走本地正则解析兜底（"明天下午3点X"句式覆盖 80% 场景）；语音明确提示需联网 | 创新 ↑、健壮性 ↑ | 中 |

## 四、语音交互深度打磨（语音优先产品的灵魂）

1. **纠错路径（最重要）**：识别结果展示后可点击编辑——卡片上的识别文本是可改的 TextInput，改完重新走 NLU（不必重录音）。错误提示从"抱歉没理解"升级为引导式："我听到的是『…』，可以试试说出时间和事情，比如…"。
2. **实时反馈分级**：录音中显示流式 ASR 中间结果（讯飞已返回中间帧，`xf_asr_stream.py` 透传到 WebSocket 即可）→ 用户边说边看到字出现，信任感倍增，也是 demo 大亮点。
3. **确认机制分险级**：添加=轻确认（卡片+可撤销，不打断）；删除/修改=强确认（弹窗）；批量=逐项可勾选。原则：**低风险不打断，高风险必确认**。
4. **多轮补全**：缺时间→追问；缺标题→追问"做什么？"。上限两轮，避免啰嗦。
5. **中文时间理解持续加固**：建一个时间表达测试集（"下周三""月底""国庆假期""大后天晚上"等 50+ 条）作为 NLU 回归测试，每次改 prompt 跑一遍——这同时是"代码质量 40%"的展示素材。

## 五、技术架构与代码质量建议（对应评分 40%）

| 方向 | 现状问题 | 建议 |
|------|---------|------|
| **状态管理** | `CalendarScreen.tsx` 460+ 行，状态/逻辑/UI 混杂 | 抽 `useVoiceSession`、`useEvents` 两个自定义 hook；事件数据收敛到一个 store（zustand 或 Context），组件只订阅 |
| **错误处理** | 散落的 try/catch + 字符串提示 | 统一 `AppError` 类型（网络/ASR/NLU/DB 四类），UI 按类型给不同引导；后端统一异常→结构化错误码 |
| **测试** | 后端 14 个测试，前端仅 service 层 | 加：NLU 时间表达回归集（上面第四节）、`calendarIntentService` 意图分支全覆盖、SQLite 迁移测试；CI 用 GitHub Actions 跑 lint+test，README 挂徽章 |
| **数据层** | schema 演进无迁移机制 | `storageService` 加版本号 + 迁移函数（P0-3 加字段时就需要） |
| **后端** | WebSocket 错误分支多、TTS 缓存无过期 | `/ws/voice` 协议补充心跳与显式 close code；TTS 缓存加 TTL；补 `xf_asr_stream` 的单测（mock websockets） |
| **可观测性** | 日志够用但无指标 | 加每阶段耗时统计端点 `/api/stats`，答辩时直接展示"全链路 P50 延迟"数据，非常加分 |
| **PR 纪律** | — | 每个 P0/P1 功能 = 1-3 个小 PR，描述带"功能/思路/测试方式"，持续提交节奏本身就是评分项 |

## 六、演示与答辩策略（对应评分 20%）

### Demo 视频叙事（按"痛点→魔法→信任→深度"四幕）

1. **痛点 30s**：双手占用场景（做饭/开车），手动输入日程多麻烦。
2. **魔法 60s**：一句话多事件拆分——"明早健身、下午三点开会、晚上聚餐"→ 三张卡片飞入 → 批量确认。这是全片最大的"哇"点，放最前面。
3. **信任 60s**：解析卡片透明展示 → 故意说一句含糊的（演示追问补全）→ 删除强确认 → 撤销演示。传达"AI 不乱来"。
4. **深度 45s**："看看这周安排"→ 周议程 + 日程洞察播报；提醒通知弹出实拍。
5. **技术 30s**：架构图一页：流式 ASR 全链路 1-3 秒、本地 SQLite 隐私、NLU 回归测试集与 CI 截图。

### 答辩弹药

- **延迟数据化**：`/api/stats` 的真实 P50 数字 > 形容词。
- **隐私架构**：后端无状态是被问"数据安全"时的标准答案。
- **质量证据**：时间表达回归集 + CI 徽章 + 小步 PR 记录，直接回应"代码质量 40%"。
- **路线图**：被问"后续规划"时拿 P2 列表回答，显得产品思考完整。

## 七、推荐落地顺序

```
第一波（核心闭环）：P0-5 解析反馈卡片 → P0-4 撤销 → P0-2 范围查询
第二波（能力跃迁）：P0-1 多事件拆分 → P1 多轮补全/指代 → 日程洞察
第三波（信任补全）：P0-3 提醒通知（重打包窗口期做）→ 搜索 → 周视图
第四波（质量与演示）：架构重构(hooks/store) → NLU 回归集 + CI → Demo 视频重录
持续：每个功能小步 PR，保持提交节奏
```

> 总原则：**先把"说一句话 → 看懂 → 可撤销 → 能提醒"的信任闭环做满，再扩能力边界。** 语音产品的胜负手不在功能数量，而在用户敢不敢把日程交给它。
