## 问题复盘

### 问题 1：Android 禁止明文 HTTP 请求

**现象**：前端文本输入一直提示"文本解析失败，请确认后端已启动"，但后端确认正常运行。

**根因**：AndroidManifest.xml 中 usesCleartextTraffic 使用了 Gradle 变量替换，但 build.gradle 的 debug buildType 里没有定义 manifestPlaceholders，导致变量未被替换为 true。Android 默认禁止明文 HTTP（http://），只允许 HTTPS。

**修复**：
1. build.gradle 的 debug buildType 添加 manifestPlaceholders
2. AndroidManifest.xml 中 usesCleartextTraffic 硬编码为 true
3. 新增 res/xml/network_security_config.xml 显式允许 10.0.2.2 域名的明文流量

**教训**：RN 项目中如果后端用 http://（非 HTTPS），必须显式配置 Android 的 cleartext 权限。不能只依赖 Gradle 变量替换，最好用 network_security_config.xml 做双重保障。

---

### 问题 2：build.gradle 被 PowerShell 5 写入 UTF-8 BOM

**现象**：Gradle 编译报错 Unexpected character at line 1, column 1。

**根因**：使用 PowerShell 5 的 Set-Content -Encoding UTF8 写入文件时会自动添加 BOM（字节顺序标记 EF BB BF），Gradle 不认 BOM。

**修复**：用 System.IO.File 的 ReadAllBytes 去掉前 3 字节 BOM。

**教训**：Windows 下写文件务必使用 PowerShell 7 的 Out-File -Encoding utf8NoBOM 或 System.IO.File.WriteAllText，永远不要用 PowerShell 5 的 Set-Content -Encoding UTF8 写 Gradle/Java 配置文件。

---

### 问题 3：后端 NLU 返回 intent=null（静默异常）

**现象**：后端 /api/nlu/parse 返回 200 但所有字段为 null。单独运行 Python 测试脚本则正常返回。

**根因**：nlu.py 的 parse_intent 函数中 except Exception 捕获了所有异常并静默返回 intent=None。所有错误都被吞掉，没有任何日志，导致极难排查。

**修复**：
1. 将 except Exception 改为 except Exception as e
2. 添加 logging.error 输出异常类型和信息
3. 在关键步骤添加 logging.info 输出 DeepSeek 原始返回

**教训**：永远不要用空的 except Exception pass，至少要记录日志。开发阶段 catch 块应该显示具体错误，而不是笼统的"请确认后端已启动"。

---

### 问题 4：react-native-sqlite-storage 原生模块未注册

**现象**：app 启动后显示 SQLite not loaded。

**根因**：react-native-sqlite-storage v6 使用旧的 ReactPackage 方式注册原生模块。React Native 新架构（0.82+）的 autolink 只自动注册 TurboModule，不会自动将旧的 SQLitePluginPackage 加入 PackageList。查看 autolinking 生成的 autolinking.cpp，确认其中只有 safeareacontext，完全没有 sqlite。

**修复**：在 MainApplication.kt 中手动添加 SQLitePluginPackage：

```kotlin
import org.pgsqlite.SQLitePluginPackage

// 在 packages.apply 块中：
add(SQLitePluginPackage())
```

**教训**：旧版 Native Module 库（非 TurboModule）在新架构 RN 下可能需要手动注册。遇到 not loaded 类错误时，先检查 autolink 的生成文件确认模块是否真的被链接。选型时优先考虑已适配新架构的库。

---

### 问题 5：Metro 端口冲突和残留进程

**现象**：run-android 时反复提示 Another process is running on port 8081。

**根因**：多次调试过程中遗留了大量未清理的 node 进程。

**修复**：通过 Get-Process 定位所有 Metro/React Native 相关进程并 Stop-Process。

**教训**：调试中断后要清理残留进程再重新启动。可以用 npx react-native start --reset-cache 强制重启 Metro。

---

### 问题 6：环境变量覆盖 .env 配置

**现象**：.env 文件中 DEEPSEEK_API_KEY 为 sk-4ea79...，但 config.py 读到的是 sk-6c56f54...。

**根因**：系统环境变量中已设置 DEEPSEEK_API_KEY，pydantic-settings 的优先级是环境变量大于 .env 文件，所以 .env 里的值被覆盖了。

**教训**：pydantic-settings 的配置优先级为环境变量 > .env 文件 > 默认值。团队协作时注意区分本地 .env 和全局环境变量，避免 key 不一致。
