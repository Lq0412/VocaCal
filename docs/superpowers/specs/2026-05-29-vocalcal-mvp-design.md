# VocaCal MVP 设计方案

> 策略：方案 A（最小可行）+ 讯飞方言免切（低成本亮点）
> 时间约束：2.5 天（5/29 下午 ~ 5/31 23:59）

---

## 1. 产品定位

**一句话**：用语音快速管理日程的手机 App，说出 "明天下午三点开会" 即可完成事件添加。

**目标用户**：需要快速记录日程的通用用户，核心场景是"不方便打字时快速添加/查看/删除日程"。

**不做的事**：云同步、多用户、重复日程、地点事件、日程分享。

---

## 2. 核心用户需求

基于竞品分析（Google Calendar Voice / Apple Siri / 国内语音助手）和中文语音场景调研，提炼出：

### 2.1 用户真的怎么说

用户不会说完整句，而是高度压缩的表达：

| 模式 | 示例 |
|------|------|
| 时间在前 | "明天下午三点开会"、"下周一上午十点项目评审" |
| 事件在前 | "提醒我开会，明天下午三点"、"牙医预约，周五上午" |
| 超简短 | "后天三点开会"、"这周末回家"、"月底交报告" |

### 2.2 最大的痛点

| 排名 | 痛点 | 原因 |
|------|------|------|
| 1 | **识别错误导致错误事件** | 中文同音词密集："十点"→"四点"、"周一"→"周日" |
| 2 | **相对时间解析不准** | "下周三"在不同星期几含义不同 |
| 3 | **缺少时间时不知道怎么办** | "明天开会" 没说几点 |
| 4 | **网络延迟体验差** | 语音→结果超过 3 秒就觉得坏了 |

### 2.3 信任 = 确认 + 透明

> 如果语音识别错了时间而用户不知道，比没有这个 App 还糟。

核心设计原则：**每一步都让用户看到系统理解了什么**。

## 3. 功能范围

### P0 — 必须有（Day 1-2）

| # | 功能 | 验收标准 |
|---|------|----------|
| 1 | 语音添加事件 | 说 "明天下午三点开会" → 识别文字显示 → 事件卡片预览 → TTS 确认 → 保存 |
| 2 | 语音查询事件 | 说 "明天有什么安排" → 按时间序列出事件 → TTS 播报 |
| 3 | 语音删除事件 | 说 "把明天的开会删掉" → 确认弹窗 → TTS 确认 → 删除 |
| 4 | 识别文字可视化 | ASR 结果立即在屏幕显示，用户可确认或重说 |
| 5 | TTS 语音反馈 | 每次操作都有语音回复 + 屏幕文字同步显示 |
| 6 | 日历 UI | 月视图 + 选中日期的事件列表 |

### P1 — 应该有（Day 2-3）

| # | 功能 | 说明 |
|---|------|------|
| 7 | 删除前确认 | 弹窗 "确定要删除明天下午 3 点的开会吗？" |
| 8 | 缺少时间的智能默认 | "明天开会" → 默认全天事件或上午 9 点，回复中说明 |
| 9 | 失败时引导 | "没听清，试试说：明天下午三点开会" |
| 10 | 查询结果按时间排序 | "明天有 3 个日程：上午 10 点评审，下午 3 点开会，晚上 7 点健身" |

### 亮点 — 方言免切（Day 3，几乎白送）

| # | 功能 | 说明 |
|---|------|------|
| 11 | 讯飞方言免切 | ASR 参数 `domain: "xfime-mianqie"`，支持 23 种方言无需切换。Demo 时说方言直接惊艳 |

### 明确不做

- ❌ 修改日程（引导用户先删再加）
- ❌ 推送提醒（Local Notification 可选做，不作为核心）
- ❌ 重复日程、日历分享、云同步
- ❌ 多轮对话

## 4. 数据模型

### 4.1 事件（Event）

```typescript
interface CalendarEvent {
  id: number;            // 自增主键
  title: string;         // 事件标题，如 "开会"
  date: string;          // 日期 "YYYY-MM-DD"
  time: string | null;   // 时间 "HH:MM"，null 表示全天事件
  note: string | null;   // 备注（语音原文）
  created_at: string;    // 创建时间 ISO 8601
  updated_at: string;    // 更新时间 ISO 8601
}
```

### 4.2 NLU 请求/响应

```typescript
// DeepSeek 解析结果
interface NLUResult {
  intent: "ADD_EVENT" | "DELETE_EVENT" | "QUERY_EVENT";
  title: string | null;       // 事件标题
  date: string | null;        // "YYYY-MM-DD"
  time: string | null;        // "HH:MM"
  raw: string;                // 原始语音文本
}
```

### 4.3 后端 API 响应

```typescript
// POST /api/voice/process 响应
interface VoiceProcessResponse {
  text: string;               // ASR 识别文本
  intent: string;             // 解析出的意图
  event: NLUResult | null;    // 解析出的事件信息
  reply_text: string;         // TTS 前的回复文本
  reply_audio: string;        // Base64 编码的 TTS 音频
}
```

## 5. 交互流程设计

### 5.1 添加事件

```
用户按住麦克风按钮
        │
        ▼ [录音动画：脉冲效果]
用户说："明天下午三点开会"
        │
        ▼
ASR 返回文字 → 屏幕显示："你说的是：明天下午三点开会"
        │
        ▼
NLU 解析 → 屏幕显示事件卡片预览：
  ┌─────────────────────────┐
  │ 📅 5月30日（周六）       │
  │ 🕐 下午 3:00            │
  │ 📝 开会                  │
  └─────────────────────────┘
        │
        ▼
TTS 播报："已添加 5月30日周六下午3点开会"
        │
        ▼
事件保存到 SQLite → 日历 UI 刷新
```

### 5.2 查询事件

```
用户："明天有什么安排"
        │
        ▼
ASR → NLU 解析为 QUERY_EVENT { date: "2026-05-30" }
        │
        ▼
查询 SQLite → 按时间排序
        │
        ├── 有事件 → TTS："明天有 3 个日程：上午 10 点项目评审，
        │           下午 3 点开会，晚上 7 点健身"
        │           屏幕同步显示事件列表
        │
        └── 无事件 → TTS："明天暂时没有安排"
```

### 5.3 删除事件

```
用户："把明天的开会删掉"
        │
        ▼
ASR → NLU 解析为 DELETE_EVENT { title: "开会", date: "2026-05-30" }
        │
        ▼
SQLite 模糊匹配 → 找到 1 个匹配事件
        │
        ▼
弹窗确认："确定要删除 5月30日（周六）下午3点的「开会」吗？"
  [取消]  [确认删除]
        │
        ▼ (确认)
删除 → TTS："已删除明天下午3点的开会"
```

### 5.4 错误处理

```
场景                    → App 行为
───────────────────────────────────────────────────
ASR 返回空/听不清        → "没听清，请再说一次" + 麦克风回到待命
NLU 无法识别意图         → "抱歉没理解，试试说：明天下午三点开会"
删除时未找到匹配事件     → "没找到明天的「开会」，请确认事件名称"
网络异常                → "网络连接失败，请检查网络后重试"
```

## 6. 技术架构

### 6.1 整体

```
┌─────────────────────────────────────────────────┐
│                  React Native App                │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ 语音录制  │  │ 日历 UI  │  │ 事件列表/确认  │  │
│  └────┬─────┘  └────▲─────┘  └───────▲───────┘  │
│       │              │                │          │
│       │    ┌─────────┴────────────────┘          │
│       │    │         SQLite (本地存储)            │
│       │    └─────────────────────────▲           │
└───────┼──────────────────────────────┼───────────┘
        │ (音频流)                      │ (REST)
        ▼                              │
┌───────────────────────────────────────┴───────────┐
│                FastAPI 后端                         │
│  ┌─────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ 讯飞 ASR    │  │ DeepSeek NLU │  │ 讯飞 TTS  │ │
│  │ (语音→文字) │  │ (意图+实体)  │  │ (文字→语音)│ │
│  └─────────────┘  └──────────────┘  └───────────┘ │
└───────────────────────────────────────────────────┘
```

后端无状态，所有事件数据在 App 端 SQLite。

### 6.2 前端

| 技术 | 选型 | 理由 |
|------|------|------|
| 框架 | React Native + TypeScript | 跨平台，3 天可出原型 |
| 导航 | React Navigation | RN 标准导航方案 |
| 日历 UI | `react-native-calendars` | 成熟组件，自带月视图 |
| SQLite | `react-native-sqlite-storage` | RN 侧主流 SQLite 库 |
| 音频录制 | `react-native-audio-recorder-player` | 录制 WAV/PCM |
| HTTP | `axios` | API 调用 |

### 6.3 后端

| 技术 | 选型 | 理由 |
|------|------|------|
| 框架 | FastAPI | 异步支持好，开发快 |
| ASR | 讯飞语音听写 WebSocket API | 中文最强，支持方言免切 |
| TTS | 讯飞在线语音合成 API | 与 ASR 统一 SDK |
| NLU | DeepSeek Chat API | 理解中文口语强，价格低 |
| 音频处理 | `python-multipart` | 接收 App 上传的音频文件 |

### 6.4 关键技术参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 讯飞 ASR eos | 5000ms | 用户思考停顿不会过早截断（默认 2000ms 太短） |
| 讯飞 ASR domain | `xfime-mianqie` | 方言免切模式，支持 23 种方言 |
| DeepSeek temperature | 0.1 | NLU 解析需要确定性输出 |
| TTS 语速 | 中等 | 清晰自然 |

## 7. DeepSeek NLU Prompt 设计

```python
SYSTEM_PROMPT = """你是一个日历事件解析助手。根据用户的语音输入，提取意图和事件信息，以 JSON 返回。

支持的意图：
- ADD_EVENT: 添加事件
- DELETE_EVENT: 删除事件
- QUERY_EVENT: 查询事件

返回格式：
{
  "intent": "ADD_EVENT | DELETE_EVENT | QUERY_EVENT",
  "title": "事件标题（简洁，如'开会''健身'）",
  "date": "YYYY-MM-DD（解析后的绝对日期）",
  "time": "HH:MM（24小时制，如 15:00）",
  "raw": "原始语音文本"
}

规则：
1. 当前日期：{today}（{weekday}）
2. 解析相对时间：明天=后一天，后天=后两天，大后天=后三天
3. "下周X" = 下一个周X（从当前周之后算起）
4. "这周末" = 最近一个周六和周日
5. 模糊时间默认值：上午=09:00，下午=15:00，晚上=19:00，中午=12:00
6. 如果用户没说时间，time 设为 null（表示全天事件）
7. 查询意图不需要 title 和 time
8. 只返回 JSON，不要其他文字"""
```

## 9. API 设计

### `POST /api/voice/process`

接收音频，返回 ASR 文本 + NLU 解析 + TTS 音频回复。

**请求**：`multipart/form-data`
- `audio`: 音频文件（WAV/PCM，16kHz，16bit，单声道）

**响应**：
```json
{
  "text": "明天下午三点开会",
  "intent": "ADD_EVENT",
  "event": {
    "title": "开会",
    "date": "2026-05-30",
    "time": "15:00"
  },
  "reply_text": "已添加5月30日周六下午3点开会",
  "reply_audio": "<base64>"
}
```

**错误响应**：
```json
{
  "text": "明天...",
  "intent": null,
  "event": null,
  "reply_text": "抱歉没理解，试试说：明天下午三点开会",
  "reply_audio": "<base64>"
}
```

### `POST /api/nlu/parse`

纯文本 NLU 解析（调试用，不走 ASR/TTS）。

**请求**：
```json
{ "text": "明天下午三点开会" }
```

**响应**：
```json
{
  "intent": "ADD_EVENT",
  "title": "开会",
  "date": "2026-05-30",
  "time": "15:00",
  "raw": "明天下午三点开会"
}
```
